DRILL 06 — Traffic Analysis with tcpdump
Topic: Traffic Analysis
Repeat daily until instant.

--- READ ---
sudo tcpdump -i enp1s0 -n              # all traffic, no DNS resolution, raw and unfiltered
sudo tcpdump -i enp1s0 -n icmp         # filter to just ICMP (ping) traffic
sudo tcpdump -i enp1s0 -n tcp port 22  # filter to just SSH traffic

--- WHAT TO LOOK FOR ---
ICMP echo request / echo reply pairs        = ping working end to end
echo request with NO matching echo reply    = packet sent, never answered — proof something dropped it
SYN with no SYN-ACK                          = TCP equivalent — connection attempt, no response
seq numbers                                  = match each request to its specific reply, pinpoint exactly which packets vanished

--- KEY PRINCIPLE ---
tcpdump never lies about what crossed YOUR interface. Application logs can be wrong, missing,
or misleading — tcpdump shows the literal wire truth, no interpretation.

--- LIMITATION TO REMEMBER ---
A capture on YOUR machine alone proves what left your interface. It does NOT prove what
happened after — could be dropped by your own firewall, a router in between, or the
destination's firewall. To prove exactly where a packet died, you need captures on BOTH ends.

--- BREAK ---
sudo iptables -I OUTPUT -d 8.8.8.8 -p icmp -j DROP   # silently drop outbound ICMP
ping -c 2 8.8.8.8                                      # 100% packet loss
sudo tcpdump -i enp1s0 -n icmp                         # confirm: requests leave, no replies ever appear

--- FIX ---
sudo iptables -D OUTPUT -d 8.8.8.8 -p icmp -j DROP
ping -c 2 8.8.8.8                                      # confirm replies return

--- PRODUCTION SCENARIO ---
Two teams disagree: sender's app logs claim success, receiver claims nothing arrived.
Step 1: tcpdump on the sending machine during the exact request
Step 2: look for the equivalent of SYN/echo-request leaving
Step 3: look for SYN-ACK/echo-reply coming back
Step 4: state precisely what the evidence shows — "request left, no reply returned" —
        do NOT overclaim "it never reached them," only claim what you can prove from your side
Step 5: if you need to prove exactly where it died, you need a capture on the receiving end too

--- BANNER GRABBING: WHAT A SERVICE VOLUNTEERS BEFORE ANY LOGIN ---

CONCEPT:
A banner is information a service announces unprompted, the instant a connection opens,
before any authentication or real interaction. Pure reconnaissance for an attacker — zero
exploitation needed, the service hands it over for free.

GRAB AN SSH BANNER:
echo "" | nc -w 2 localhost 22
Reveals: exact SSH implementation, version, and (on Ubuntu) the distro patch suffix.
Why it matters: a published vulnerability for a specific version gives attackers a
pre-built target list just by scanning for this exact string.

GRAB AN HTTP/nginx BANNER:
curl -I http://localhost
Look at the "Server:" header. Default nginx reveals full version + OS in this header.

MITIGATION — nginx (config-controllable):
server_tokens off;   in /etc/nginx/nginx.conf
Removes version number from the Server header. Verify with: curl -I, check Server: line.
Always back up config first, test with `nginx -t`, then reload — never edit blind.

MITIGATION — SSH (PARTIALLY controllable, honest limitation):
VersionAddendum none   in /etc/ssh/sshd_config
This does NOT remove the core "SSH-2.0-OpenSSH_X.X" string — that's part of the protocol
itself. On Ubuntu specifically, it also does NOT remove the distro suffix
(e.g. "Ubuntu-3ubuntu13.16") — that's compiled into the binary at package build time,
not something sshd_config can reach. Verified directly: even with VersionAddendum
correctly set and confirmed active via `sudo sshd -T`, the banner is unchanged.

REAL LESSON: not everything a service leaks is fixable from its config file. Some
information is baked in at compile time. The professional response isn't "configure it
away" when that's not possible — it's knowing the limitation exists and compensating
elsewhere: stay patched, scope access tightly with firewall rules (Topic 6), don't expose
SSH to the internet unnecessarily. A banner you can't hide is mitigated by reducing who
can reach it and ensuring the version itself has no known exploitable vulnerabilities.

VERIFY CURRENT LIVE CONFIG (not just the file on disk):
sudo sshd -T | grep -i versionaddendum
This shows what's actually loaded and active right now, not what the file claims.
